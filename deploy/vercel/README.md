# Vercel deployment

EquityLens deploys the Next.js web app and FastAPI API from one Git repository
into one Vercel Project through [Vercel Services](https://vercel.com/docs/services).

| Service | Root | Framework | Public traffic |
|---|---|---|---|
| `web` | `frontend/` | Next.js | Every route outside `/api/v1/*` |
| `api` | `backend/` | FastAPI | `/api/v1/*` |

The root [`vercel.json`](../../vercel.json) builds both services independently.
A Service Binding injects the private API URL as `BACKEND_URL` into the web
runtime. The API derives the public Workflow origin from Vercel's automatic
`VERCEL_URL`. Each Preview deployment therefore calls its matching Preview
routes without a circular Service Binding.

## 1. Prepare managed dependencies

Create these resources before opening the Deploy Button:

- PostgreSQL with the `pgvector` extension and a production `DATABASE_URL`;
- a private Vercel Blob store and its `BLOB_READ_WRITE_TOKEN`;
- an OpenAI API key;
- a Google OAuth Web client ID;
- four independent application secrets of at least 32 random characters.

Generate each application secret locally:

```bash
openssl rand -hex 32
```

Use separate values for `SECRET_KEY_ACCESS_API`, `GUEST_SIGNING_SECRET`,
`QUOTA_HASH_SECRET`, and `INTERNAL_JOB_SECRET`.

## 2. Deploy the Services Project

Use the single Deploy Button in the root [`README.md`](../../README.md), or
import the repository through the Vercel dashboard with the repository root as
the Project Root Directory. Vercel reads the root `vercel.json` and creates both
services in one deployment.

The Deploy Button supplies these public profile defaults:

```dotenv
DEPLOYMENT_TARGET=vercel
OBJECT_STORAGE_PROVIDER=vercel_blob
JOB_BACKEND=vercel_workflow
DOCUMENT_PARSER=managed
```

Provide the shared environment variables through Vercel:

- `DATABASE_URL`
- `SECRET_KEY_ACCESS_API`
- `OPENAI_API_KEY`
- `OPENAI_ORGANIZATION`
- `OPENAI_BASE_URL` (optional)
- `LLM_API_KEY` (optional Chat Completions provider override)
- `LLM_BASE_URL` (optional Chat Completions provider override)
- `LLM_STRUCTURED_OUTPUT_METHOD` (defaults to `json_schema`)
- `TAVILY_API_KEY` (optional; blank uses rate-limited keyless search)
- `CHAT_WEB_SEARCH_PROVIDER` (defaults to `tavily`)
- `CHAT_TAVILY_SEARCH_DEPTH` (defaults to `basic`)
- `CHAT_TAVILY_MAX_RESULTS` (defaults to `5`)
- `FIRST_SUPERUSER`
- `FIRST_SUPERUSER_PASSWORD`
- `GOOGLE_CLIENT_ID`
- `NEXT_PUBLIC_GOOGLE_CLIENT_ID` (same value as `GOOGLE_CLIENT_ID`)
- `BLOB_READ_WRITE_TOKEN`
- `GUEST_SIGNING_SECRET`
- `QUOTA_HASH_SECRET`
- `INTERNAL_JOB_SECRET`
- `SEC_USER_AGENT`
- `MARKET_DATA_PROVIDER`
- `RESEARCH_MODEL`
- `SUPPLY_CHAIN_GRAPH_MODEL_OVERRIDE` (optional)

`BACKEND_URL` is generated by the web-to-API Service Binding. `VERCEL_URL` is a
Vercel system variable used to derive the three Workflow endpoints. The
explicit Workflow trigger variables remain supported as legacy overrides.
`FRONTEND_URL` is optional because same-origin validation reads Vercel's
forwarded deployment origin. `MANAGED_PARSER_API_KEY` remains reserved for a
future parser integration.

The API derives these private Workflow trigger endpoints from the web binding:

- `/api/internal/workflows/company-intelligence`
- `/api/internal/workflows/supply-chain-graph`
- `/api/internal/workflows/filing-index`

`LLM_API_KEY` defaults to `OPENAI_API_KEY`, and `LLM_BASE_URL` defaults to
`OPENAI_BASE_URL`. A DeepSeek-backed generation profile can set
`LLM_BASE_URL=https://api.deepseek.com`,
`LLM_STRUCTURED_OUTPUT_METHOD=json_mode`, and
`CHAT_WEB_SEARCH_PROVIDER=tavily`. Tavily supplies web search and OpenAI supplies
the filing embeddings used by research chat.

The guest secret signs short-lived anonymous research assertions. The internal
secret starts a durable Workflow run and signs each idempotent FastAPI step
request using only the database job ID.

## 3. Apply the database schema

Run Alembic against the production PostgreSQL database before testing the API:

```bash
cd backend
DATABASE_URL="postgresql://..." uv run alembic upgrade head
```

The first migration enables `pgvector`. The database user needs permission to
run `CREATE EXTENSION IF NOT EXISTS vector`.

## 4. Private research evidence storage

Connect the private Vercel Blob store to the Services Project. Set its generated
`BLOB_READ_WRITE_TOKEN` in Production and Preview environments. The
graph collector writes compressed official-source artifacts with
`access="private"`; the public API exposes capped evidence excerpts and source
URLs while the raw artifacts stay behind the server-side token.

Bounded web evidence collected for research chat uses the private `chat-web/`
namespace. The public response carries exact excerpts and source URLs, while
the full fetched artifact remains accessible through the API's server-side
token.

Keep these defaults aligned across deployments so cached graph snapshots remain
reproducible:

```dotenv
SUPPLY_CHAIN_GRAPH_SCHEMA_VERSION=supply-chain-graph.v1
SUPPLY_CHAIN_GRAPH_PROMPT_VERSION=supply-chain-graph.2026-07-14
SUPPLY_CHAIN_GRAPH_MAX_NODES=40
SUPPLY_CHAIN_GRAPH_SOURCE_LIMIT=24
SUPPLY_CHAIN_GRAPH_SOURCE_BYTES=8000000
SUPPLY_CHAIN_GRAPH_EVIDENCE_TOKEN_BUDGET=100000
GRAPH_ARTIFACT_PREFIX=supply-chain
```

`SUPPLY_CHAIN_GRAPH_MODEL_OVERRIDE` selects a graph-specific model. An empty
value applies `RESEARCH_MODEL` to both research pipelines.

## 5. Data-source and quota constraints

Set `SEC_USER_AGENT` to an application name and monitored contact email. SEC
traffic follows the published [automated-access guidance](https://www.sec.gov/about/webmaster-frequently-asked-questions).

The current Yahoo adapter supports research and evaluation. Complete a market
data licensing review before public or commercial distribution.

The compact 10-K pipeline persists its source bytes and integrity metadata in
PostgreSQL. The supply-chain graph pipeline stores compressed official-source
artifacts in private Vercel Blob and durable stage records in PostgreSQL.
Workflow steps depend on durable records and object keys.

Research chat retrieves indexed 10-K chunks through PostgreSQL full-text search
and pgvector. When the current filing lacks an index, the API dispatches a
zero-quota Workflow through `/api/internal/workflows/filing-index`; its single
idempotent step creates or replaces the filing chunks and embeddings.

Official-source collection accepts at most 24 sources and 8 MB total under the
default profile. `SEC_USER_AGENT` remains mandatory for SEC requests. The
collector also enforces host allowlists, DNS pinning, redirect controls, content
types, decompression bounds, and per-host pacing.

Each newly accepted graph job reserves one Agent quota unit. Cached snapshots
and active jobs reuse the current result at zero cost. System and retryable
collection failures refund the reservation idempotently; a completed
insufficient-evidence result consumes the accepted unit.

Chat uses an independent quota ledger. Guests receive two messages per UTC day,
authenticated users receive ten, and accepted system failures refund the
message reservation. Guest conversations and messages expire after seven days.

## 6. Configure the public authentication origin

Set the same Google OAuth client ID in `GOOGLE_CLIENT_ID` and
`NEXT_PUBLIC_GOOGLE_CLIENT_ID`. After the first deployment, add the generated
Vercel production origin to the OAuth client's Authorized JavaScript origins.
Add selected Preview origins when Google sign-in needs Preview verification.

The web runtime validates mutations against Vercel's forwarded host and
protocol. `FRONTEND_URL` remains available as an explicit origin override for
custom proxy deployments. Public browser traffic reaches both services through
one origin, so the standard Services deployment requires no cross-origin API
configuration.

| Service | Generated binding | Shared authentication values |
|---|---|---|
| `web` | `BACKEND_URL` | `NEXT_PUBLIC_GOOGLE_CLIENT_ID`, `GUEST_SIGNING_SECRET`, `INTERNAL_JOB_SECRET` |
| `api` | System `VERCEL_URL` | `GOOGLE_CLIENT_ID`, `SECRET_KEY_ACCESS_API`, `GUEST_SIGNING_SECRET`, `INTERNAL_JOB_SECRET` |

## 7. Verify production

```bash
WEB_BASE_URL=https://equitylens.example.com \
API_BASE_URL=https://equitylens.example.com \
./scripts/smoke.sh
```

The expected endpoints are:

- Web health: `GET /api/health`
- Public dashboard: `GET /en-US/dashboard`
- Research BFF: `GET /api/research/companies/search?q=AAPL`
- Chat readiness: `GET /api/research/companies/AAPL/chat-readiness`
- API smoke alias: `GET /api/v1/health`
- API liveness: `GET /api/v1/health/live`
- API readiness: `GET /api/v1/health/ready`

Generate an AAPL graph from the English company page, open one verified
relationship, enable potential relationships, switch to Chinese, and refresh
the page. The published snapshot and quota count should remain stable.

Open the company research panel, prepare the filing index when prompted, send a
filing question, and confirm that answer text and citations arrive
incrementally. Repeat the journey as a guest and authenticated user to verify
the independent daily message limits.

## Local Vercel builds

Use the latest Vercel CLI from the repository root:

```bash
pnpm dlx vercel@latest pull --yes --environment=preview
pnpm dlx vercel@latest build
```

The root [`vercel.json`](../../vercel.json) owns Services, bindings, public
routing, and function limits. [`backend/runtime.txt`](../../backend/runtime.txt)
pins the Python runtime for the API service.
